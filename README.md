# Fortune Wheel

Мобильное приложение с колесом фортуны, где можно крутить колесо и выигрывать подарки разной редкости

## Технологии

### State Management — flutter_bloc (Cubit)

Выбран Cubit вместо полноценного Bloc, потому что в приложении нет сложных цепочек событий, и не требуется асбтрактность bloc, достаточно простых вызовов методов. Cubit проще, меньше бойлерплейта и легче читается

### Dependency Injection — get_it + injectable

get_it- лёгкий сервис-локатор, а injectable автоматически генерирует регистрацию зависимостей по аннотациям

### Модели данных — freezed + json_serializable

freezed генерирует иммутабельные классы с и sealed union типами (например, Gift.regular и Gift.coin). json_serializable добавляет сериализацию в JSON для сохранения данных

### Локальное хранилище — shared_preferences

Простое key-value хранилище для сохранения баланса монет, инвентаря подарков и состояния колеса между сессиями. Для тестового задания эта база данных избыточна

### Сетевой слой — http

Минимальный HTTP-клиент для запросов к randomnumberapi.com. Приложение получает случайные числа для определения выигрыша. Числа загружаются пачками по 5 и кэшируются, чтобы не делать запрос на каждое вращение и получать ошибку 429 (Too many requests), dio я посчитал избыточным, у нас тут нет авторизации, где он очень полезен с интерсепторами

### Анимации — flutter_animate + confetti

flutter_animate — декларативные анимации через цепочки .animate().fadeIn().slideX(). Используется для shimmer-эффектов, появления элементов и переходов. confetti - готовые конфетти-частицы для celebration-эффекта при выигрыше редких подарков, также в приложении много кастомных анимаций, сделанных через AnimationController и CustomPainter

### Звук — flutter_soloud

Аудио-движок для звуковых эффектов: щелчки колеса при вращении, звуки победы разной редкости, выстрел конфетти. SoLoud выбран за низкую задержку воспроизведения, это примитивный и нативный плагин для запуска коротких звуков, я пробоал just_audio, но он медленно воспроизводил звуки

### Локализация — intl + flutter_localizations

Встроенный механизм локализации Flutter через ARB-файлы. Поддерживаются русский и английский языки

### Графика — flutter_svg + material_symbols_icons

flutter_svg для рендеринга SVG-иконок (например, иконка колеса в навигации). material_symbols_icons — набор Material Design 3 иконок



## Архитектура

Приложение разделено на два слоя:

- **data** — модели, репозитории, источники данных (сеть и локальное хранилище)
- **view** — экраны, кубиты, виджеты

я не делал entity и парсинг моделей и не икапсулировал сериализацию по такой же причине избыточности, я в целом не люблю использовать отдельный слой domain, потому что бизнес логикой отлично справляется BLoC - business logic component

Каждая фича живёт в своей папке со своим кубитом и виджетами

## Сложные моменты
Запуск запроса идет во время кручения колеса и его freeSpin, оно будет крутиться пока не получится случайной число от сервера, если интернета нет или что-то пошло не так - покажется диалоговое окно с ошибкой

Подарки сделаны абстрактными через union классы, потому что разные подарки могут делать разные вещи, так что добавление новых механик - сильно упрощено

Подарок известен заранее, до того как колесо начнет останавливаться и чтобы не было остановки ровно по середине сектора - добавлен случайный offset с небольшим паддингом, чтобы стрелочка не оказывалась на краю и не вводила пользователя в замешательство

Редкость подарков учитывается, она является мультипликатором для рандомного числа с сервера

## Возможности для улучшения

*  unit-тесты на кубиты и репозитории, widget-тесты на ключевые экраны. Кубиты уже хорошо изолированы через абстракции, так что мокать зависимости будет просто
*  сейчас выбор подарка происходит на клиенте, а сервер только даёт случайное число. В продакшене логику выбора стоит перенести на бэкенд, чтобы исключить манипуляции
* при большом количестве подарков список может тормозить из-за множества AnimatinController уже используется, но стоит добавить ленивую загрузку из хранилища, но так как подарки стакаются, я решил этого не делать
*  поддержка планшетов и landscape-ориентации. Колесо можно масштабировать под доступное пространство, можно добавить что-то вроде main-detail вертски для компьютеров
* shared_preferences не подходит для больших объёмов данных. При росте инвентаря я бы перешел на простую нереляционную вроде isar
